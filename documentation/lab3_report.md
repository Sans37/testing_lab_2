# Лабораторная работа №3
## Benchmark для оценки производительности проекта Nady Bakery

**Дата выполнения:** 2026-03-19

## 1. Цель работы
Составить набор сценариев (benchmark) для оценки производительности выбранного объекта и получить статистику минимум по 100 испытаниям.

## 2. Объект исследования
Веб‑приложение Nady Bakery (FastAPI + PostgreSQL). В фокусе — API слоя `/api/v2/*`.

## 3. Сценарии тестирования
В рамках лабораторной подготовлены три сценария:

1. **Baseline (нормальная нагрузка)**
   - Смешанный профиль запросов: чтение каталога, категории, служебные данные, авторизация, запись тестовых данных.
   - Цель: получить опорные показатели латентности и ошибок.

2. **Поиск точки деградации**
   - Нагрузка увеличивается ступенями (шаг по конкурентности).
   - Фиксируется момент, когда p95 превышает порог или растёт error rate.

3. **Перегрузка + восстановление**
   - Сначала кратковременная максимальная нагрузка.
   - Затем серия окон с базовой конкурентностью, чтобы измерить восстановление.

## 4. Параметры, которые измеряются
- Время ответа (latency) для разных типов запросов.
- Перцентили p50 / p75 / p90 / p95 / p99.
- Гистограмма распределения задержек.
- Временной ряд латентности.
- Утилизация ресурсов контейнеров (CPU/RAM) по данным `docker stats`.

## 5. Методика
- Минимум **100 прогонов** (`--trials 100`).
- Каждый прогон по необходимости может поднимать отдельный docker‑стенд.
- Для равных условий используется один и тот же docker‑compose.

## 8. Как воспроизвести

1. Поднять тестовый стенд:
```
docker compose -f docker/docker-compose.benchmark.yml up -d --build
```

2. Установить доп. зависимости (для графиков):
```
.\.venv\Scripts\python.exe -m pip install -r requirements-benchmark.txt
```

3. Запустить benchmark:
```
.\.venv\Scripts\python.exe scripts\benchmark_lab3.py --base-url http://localhost:8000 --trials 100 \
  --compose-file docker/docker-compose.benchmark.yml \
  --docker-per-trial \
  --docker-stats nady-benchmark-api nady-benchmark-db
```

4. Посмотреть результаты в `benchmarks/<run-id>/`.

## 9. Примечания
- При долгих прогонах допустимо уменьшить `--trials` для быстрой проверки.
- При необходимости можно настроить шаг деградации через `--degradation-step` и `--max-concurrency`.
